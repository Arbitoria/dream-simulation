# 리포트: "버튼이 안 눌려요"가 왜 이렇게 오래 걸렸나

## 한 줄 요약
진짜 버그는 **CSS 한 줄**이었다. 그런데 내가 **잘못된 방법으로 "정상"이라고 확인**하는 바람에, 실제 원인이 아닌 **캐시 문제를 계속 쫓느라** 여러 번 헛돌았다.

## 진짜 원인 (실제 버그)
- 언어·성별·나이·인트로 화면은 `class="screen center"` 였고, CSS에 `.screen.center{ display:flex }` 가 걸려 있었다.
- 화면을 숨기는 `hidden` 속성(`[hidden]{display:none}`)보다 이 규칙의 **우선순위가 높아서**, 숨겨야 할 화면이 계속 `display:flex`로 **보였다.**
- 그래서 `go(1)`은 정상 실행됐지만(=클릭은 처음부터 됐음), **이전 화면이 사라지지 않고 위에 그대로 떠 있어서** "아무 반응 없음"처럼 보였다.
- **수정:** `.screen[hidden]{ display:none !important }` 한 줄.

## 왜 이렇게 오래 걸렸나 — 나의 실수 5가지
1. **검증을 엉뚱한 걸로 했다.** 화면 전환을 확인할 때 `el.hidden`(속성)만 봤다. 이 값은 *올바르게* true였다. 하지만 실제로 눈에 보이는 `display`는 여전히 `flex`였다. → 내 테스트는 매번 "정상"이라고 **거짓 통과**했다.
2. **내 테스트가 통과하니 사용자 문제를 '환경 탓'으로 돌렸다.** "코드는 되는데 니 브라우저에서 안 되네 → 캐시/옛날 탭" 이라고 단정했다. 근거 없이.
3. **캐시만 여러 번 팠다.** 버전 쿼리(`?v=`), no-cache 서버, 단일 파일 빌드… 전부 진짜 원인이 아니었다. (부수적으로 서버가 단일 스레드라 무한로딩까지 겹쳐서 더 혼란스러웠다.)
4. **실제 화면을 못 봤는데 그걸 일찍 인정하지 않았다.** 이 환경은 스크린샷이 타임아웃 나고, 클릭 도구가 이벤트를 0개 전달한다(진짜 클릭이 안 됨). 즉 나는 "보이는 결과"를 확인할 수단이 없었는데, 그걸 늦게야 인정했다.
5. **가장 단순한 가설을 안 세웠다.** "클릭해도 안 넘어감 + JS 에러 없음 + 주소 해시는 바뀜" → 이건 클릭이 아니라 **렌더/CSS 문제**를 가리키는 명백한 신호였다. 진작 `display`를 확인했어야 했다.

## 무엇이 결국 원인을 잡았나
사용자가 콘솔에서 **`go(1)` → `undefined`(에러 없음)인데 화면은 안 바뀜**을 보여준 것. 이 순간 "클릭·코드 문제가 아니라 화면이 안 숨겨지는 문제"가 확정됐고, `getComputedStyle(screen).display`를 확인하자 `flex`(=안 숨겨짐)가 바로 드러났다.

## 교훈 (→ CLAUDE.md 규칙으로 저장)
- 검증은 **눈에 보이는 최종 상태**로. 내부 플래그(`.hidden`)가 아니라 실제 `display`/렌더/텍스트로.
- 사용자가 "안 된다"는데 내 테스트가 통과하면, **내 검증 방법부터 의심**한다. 환경 탓하지 않는다.
- 증상 전부를 설명하는 **가장 단순한 가설**을 먼저 세운다.
- 볼 수단이 없으면 일찍 인정하고 **사용자를 눈으로 활용**한다(콘솔 한 줄).
